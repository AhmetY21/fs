## 2024-05-24 - CSRF and Rate Limiting Origin/IP Validation in Next.js API Routes Shared with Flutter
**Vulnerability:** Missing CSRF protection and naive IP extraction on the `/api/analyze` endpoint. The endpoint was relying solely on `x-forwarded-for` for rate limiting, which could be easily spoofed or missing, and lacked CSRF protection to ensure requests originated from the legitimate web client.
**Learning:** Next.js API routes shared between web clients and non-browser clients (like a Flutter app using Dio) need flexible security validation. Security headers like `Origin` or `Host` are not sent by default from the mobile app. Enforcing a strict check on `Origin` would break mobile clients. We must gracefully support non-browser clients by bypassing the `Origin` check if the `Origin` header is absent, while strictly checking it against the target host (derived from `x-forwarded-host` or `Host`) if it is present. Additionally, correctly extracting the client IP in a secure way requires prioritizing `request.ip` or `x-real-ip` over simply the first entry of `x-forwarded-for`, which is critical for accurate rate limiting.
**Prevention:** Implement CSRF protection in shared APIs by checking the `Origin` against the host, but conditionally apply it only when the `Origin` header exists to accommodate mobile apps. For rate limiting IP extraction, prioritize platform-populated fields (`request.ip`, `x-real-ip`) and securely parse headers using `.split(',')[0].trim()`.
