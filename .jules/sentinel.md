## 2024-04-12 - Next.js API Route Rate Limit Bypass & CSRF
**Vulnerability:** The API route extracted the client IP address strictly from the `x-forwarded-for` header, enabling trivial rate limit bypass via header spoofing. Additionally, there was no validation of `Origin` or `Referer` headers, leaving the route vulnerable to Cross-Site Request Forgery (CSRF).
**Learning:** In serverless Next.js functions, `request.ip` or platform-specific headers like `x-real-ip` are the most reliable indicators of client IP. Relying solely on `x-forwarded-for` without verifying its source allows clients to bypass rate limits by supplying arbitrary comma-separated IPs. Furthermore, APIs accepting JSON payloads must still validate the origin, as attackers can use `text/plain` form submissions to trigger CSRF if the endpoint parses the body permissively or relies on ambient authority.
**Prevention:** Always prioritize `request.ip` and `x-real-ip` before falling back to extracting the first element of `x-forwarded-for`. For CSRF protection, strictly validate `Origin` and `Referer` headers against the expected `Host`, while gracefully failing open if both are absent to support non-browser API clients. Use `new URL()` within a try-catch block for safe header parsing.
