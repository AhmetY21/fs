## 2024-05-14 - IP Spoofing in Rate Limiting
**Vulnerability:** Rate limiting implementation relied solely on the `X-Forwarded-For` HTTP header to determine the client's IP address (`const ip = (request.headers.get('x-forwarded-for') ?? '127.0.0.1').split(',')[0].trim();`). This allowed an attacker to trivially bypass rate limits by injecting a spoofed `X-Forwarded-For` header in their request.
**Learning:** Next.js App Router exposes the client's actual IP address securely via `request.ip`. Trusting client-provided headers like `X-Forwarded-For` without validating that the request came from a trusted proxy (which Next.js handles automatically in production deployments like Vercel) is a security risk.
**Prevention:** Always use `request.ip` as the primary source of truth for the client IP address in Next.js App Router route handlers. Only fall back to `X-Forwarded-For` (with appropriate validation if possible) for local development or when explicitly deployed behind a trusted reverse proxy that doesn't integrate automatically with Next.js's IP resolution.

## 2024-05-14 - Missing CSRF Protection on API Route
**Vulnerability:** The `/api/analyze` endpoint did not implement any CSRF protection, relying solely on Next.js' default routing behavior. While standard CSRF attacks typically use form submissions with cookies, API endpoints that don't validate the `Origin` or `Referer` headers can still be vulnerable to cross-origin abuse, especially if they are designed to perform state-changing actions or consume valuable resources (like Gemini API calls).
**Learning:** Next.js Route Handlers (`app/api/.../route.js`) are essentially raw HTTP endpoints and do not automatically provide CSRF protection out-of-the-box in the same way that standard Server Actions might in a framework context. Developers must explicitly implement CSRF mitigation strategies if the endpoint doesn't strictly require API keys from all callers.
**Prevention:** For API endpoints exposed to browsers (like Next.js route handlers serving a frontend), always implement Origin validation by comparing the `Origin` header (sent by browsers for cross-origin POSTs) against the `Host` header. This blocks unauthorized cross-origin browser requests while allowing legitimate server-to-server calls or same-origin requests to proceed.
