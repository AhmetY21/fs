## 2024-03-27 - [High] CSRF Protection and IP Spoofing Prevention in API Routes
**Vulnerability:** The API route was missing CSRF validation (making it susceptible to Cross-Site Request Forgery from browsers) and was blindly trusting `x-forwarded-for` for rate limiting (allowing trivial IP spoofing to bypass limits).
**Learning:** Next.js API routes serving both web and mobile clients (Flutter) face a unique challenge: browser clients automatically send `Origin` headers (which must be validated for CSRF), while mobile app clients (`Dio`) do not send `Origin` or `Host` headers by default. Strict CSRF enforcement breaks mobile clients. Also, proxy configurations (like Vercel) populate `request.ip` and `x-real-ip`, which are secure against spoofing, unlike the user-controllable `x-forwarded-for`.
**Prevention:** Implement conditional CSRF validation: verify `Origin` against `Host` or `x-forwarded-host` ONLY if both headers are present. This secures browser traffic without breaking mobile APIs. For IP extraction in rate limiting, prioritize `request.ip` and `x-real-ip` before falling back to extracting the first IP from `x-forwarded-for`.
