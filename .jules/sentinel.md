## 2024-06-25 - IP Spoofing Rate Limit Bypass
**Vulnerability:** The rate limiting logic solely relied on the `x-forwarded-for` header to identify the client IP. This header is easily spoofable by clients. By sending requests with a constantly changing `x-forwarded-for` header, an attacker could bypass the rate limit completely, leading to a potential DoS attack or abuse of the Gemini API endpoint.
**Learning:** Depending exclusively on `x-forwarded-for` for client identification is a security risk in environments where it's not strictly sanitized or validated by a trusted upstream proxy. Platforms like Vercel provide `request.ip` or `x-real-ip` which are generally safer defaults. However, fallback is necessary to avoid global rate limiting if these properties are missing.
**Prevention:** Always prioritize trusted IP properties provided by the deployment platform (like `request.ip` or the `x-real-ip` header). Use `x-forwarded-for` only as a last resort fallback, extracting the first IP in the list, and be aware of its spoofability in certain environments.
